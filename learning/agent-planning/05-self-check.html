Chapter 05

自测 & 辨析

前四章建立了七种范式、一条「承诺↔适应」轴，以及推理模型把这条轴整体挪动的前沿。这一章不教新东西——用三层梯度的题逼出主动回忆，再用跨章场景题检验一件事：换到陌生场景，你还选得对范式吗？

共约 17 题，分三层梯度：概念层（对应 01）测回忆，原理层（对应 02–04）测机制理解，应用判别层（跨 01–04）测迁移——给你陌生场景，逼你在多个范式间做选择并讲清理由。所有答案集中在页面最底部一个折叠块里。先把答案写在纸上或编辑器里，写完再展开对照；直接点开等于把全教程再读一遍，retrieval 的价值归零。

A概念层（对应 01 基础）

  1. 用一句话说出「承诺↔适应」轴的两端各自买到什么、各自付出什么代价。→ §1.2
  2. 任务分解的"粒度"旋钮拧得太细、拧得太粗，各自触发什么问题？→ §1.3
  3. 三大家族分别回答哪三个不同的问题？各举一个代表范式。→ §1.4
  4. ReAct 的一轮由哪三个部分组成？哪一步会让生成暂停去调真实工具？→ §1.5

B原理层（对应 02–04）

  1. ReWOO 相比 ReAct 省约 5 倍 token，省的到底是什么？用一句话点破，不许答"调用更少"。→ §2.3
  2. Plan-and-Solve 和 Plan-and-Execute 的本质区别是什么？一个是什么、另一个是什么。→ §2.2
  3. LLMCompiler 的"任务获取单元"借鉴了 CPU 的什么机制？它在什么情况下必须停下来重新编译？→ §2.4
  4. ToT 的"状态评估器"是一个单独训练的 critic 吗？它具体怎么给状态打分？→ §3.2
  5. Reflexion 说它"零权重更新"——具体指什么？那段教训最后如何影响下一次行为？→ §3.3
  6. 为什么说 2026 年"给 LLM 外面套显式规划器"常常是净负？用"冗余/冲突操作"解释机制。→ §4.3
  7. test-time compute 把什么搬进了模型内部？它没有替代掉的是哪部分能力？→ §4.1

C应用判别层（跨 01–04 · 选范式并说理由）

这一层是本教程的"综合项目替身"。每题给一个陌生场景，要求选出范式 + 讲清为什么不选其余。光说选谁不算过关，理由才算。

  1. 场景：「同时查 50 个城市的天气，汇成一张对比表，要快。」选哪个范式？为什么不是"一次 reasoning model 调用搞定"？
  2. 场景：「多跳研究问答，每一步的发现决定下一步查什么，且工具偶尔返回意外结果。」选哪个？为什么 ReWOO 在这里会栽？
  3. 场景：「解一道逻辑谜题，有明确对错、且存在很多条候选解题路径。」选哪个？它比单纯让模型"多想一会"强在哪？
  4. 场景：「写代码通过一套隐藏单元测试，可反复提交。」选哪个？为什么单纯"失败就重写一遍"往往不够？
  5. 场景：「把一份结构清晰的需求文档，总结成 3 段话。」加不加显式规划器？为什么？
  6. 综合场景：「自动复现一篇论文的实验」这条流水线（读论文 → 找代码 → 跑通环境 → 复现多组实验 → 写差异报告），按 §4.3 判据切成段，哪几段靠模型内部规划、哪几段上显式范式（点名）？最危险的"两层规划相撞"会出现在哪段？

进阶挑战 · 刚好够不着

给一个范式"判死刑"

本教程一直在讲每个范式何时合适。反过来想：为 ReWOO、ToT、Reflexion 各写一个"绝对不该用它"的具体场景，并说清触发的是该范式哪条固有代价。能为一个工具准确划出禁区，才算真懂了它的边界。

提示（卡住再展开）

从每个范式的"主要代价"那一格反推。ReWOO 的死穴是蒙眼承诺——所以"工具结果高度不可预测、且一步错则全盘错"的场景（如依赖实时报价做连续决策）绝对别用。ToT 的死穴是调用量爆炸 + 需要可评估状态——所以"开放式、没有清晰对错信号、且对延迟/成本敏感"的场景（如实时客服闲聊）别用。Reflexion 的死穴是需要可重试 + 奖励信号——所以"一次性、不可重来、或根本无法判定成败"的任务（如发一封不可撤回的对外邮件）别用。

A · 概念层

  1. 左端"提前承诺"：买到可并行 + 省 token，代价是计划建立在假设的世界状态上、意外无法纠偏。右端"临场适应"：买到每步用最新信息纠错，代价是串行 + token/延迟随轨迹线性增长。
  2. 太细 → 过度分解：步骤数爆炸，延迟和成本飙升。太粗 → 子任务仍然太难，模型一步做不完。粒度本身是要权衡的旋钮，没有万能值。
  3. ① 任务分解："怎么把目标拆成步骤、按什么顺序执行？"（ReAct / Plan-and-Execute / ReWOO / LLMCompiler）。② 多计划选择："生成多条候选路径，怎么评估并选最好的？"（ToT / LATS）。③ 反思与提炼："上次失败了，怎么把教训用上让下次更好？"（Reflexion）。
  4. 三部分：Thought（推理）→ Action（行动）→ Observation（观察）。Action 这一步会让生成暂停，外部真正执行工具，把真实结果作为 Observation 喂回。

B · 原理层

  1. 省的是"不再把 observation 反复重新编码进推理"。ReAct 每步都把累积的全部历史重喂进 LLM，观察被一遍遍重编码；ReWOO 规划器只在开头付一次费、观察永不回流。工具调用次数和 ReAct 同量级——省的不是调用次数，是重新编码。
  2. Plan-and-Solve 是一篇论文，本质是一次 zero-shot 的 prompt 改写（"先拟计划再执行"），计划和求解在同一次生成里完成，无第二个模型。Plan-and-Execute 是 LangChain 的工程架构：强 planner 跑一次 + 廉价 executor 跑 N 次 + 每步后一道 replan 闸门。一个是改 prompt，一个是拆成两个角色 + 失败回到 planner。
  3. 借鉴 CPU 的指令获取 / 乱序执行——依赖就绪的任务立刻派发，独立任务并发。当依赖关系只有运行时才知道（数据相关的分支，如"由 A 的结果决定下一步查 B 还是 C"）时，它无法预先编出整张 DAG，必须停下来重新编译。
  4. 不是。评估器就是同一个 LLM 给自己生成的状态打分——做一小段前瞻后贴 sure/likely/impossible 标签，或把多个状态摆一起投票。没有额外训练的模型或参数，评判力全来自提示。
  5. "零权重更新"指不做任何微调，模型参数一个不动。路径：失败轨迹 + 奖励 → 写成自然语言教训 → 存进长期记忆 → 下一次同一任务时拼到 prompt 开头，靠改写输入上下文（而非改权重）重塑这次行为；记忆通常只留最近 1–3 条。
  6. 因为外部规划器和模型内部规划同时在拆任务、定下一步，两套规划相撞，产生冗余或冲突的操作——重复调同一工具、外层计划被内层推翻、对"下一步"给出矛盾指令。2023 年外壳是补 base model 不会多步想的缺；2026 年模型自己会想，外壳从补缺变添乱。
  7. 把"多步规划/推理"（拆步骤、排顺序、回溯）搬进了模型内部的一段长链思考。没替代的是"调真实工具拿真实结果"——内部思考不接触外部世界，也做不到真正并行。

C · 应用判别层

  1. LLMCompiler 式并行（或编排者-工人）。50 个城市的查询彼此独立，命中"大量独立可并行子任务 + 延迟压力"。不是一次 reasoning model 调用：难点不在"想"而在"同时做很多独立的事"——一条串行思考链吃不满吞吐，并行是模型内部补不上的物理缺口。
  2. ReAct（轴的适应端）。每步发现决定下一步 = 强顺序依赖 + 不确定，必须看到真实 Observation 再走下一步。ReWOO 会栽：它一次性蒙眼写完整张蓝图、观察不回流，当某步的意外结果本该让计划改道时，它改不了，只能照错蓝图走到底。
  3. Tree of Thoughts（多计划选择/搜索）。"多条候选路径 + 明确对错"正是树搜索的主场：生成多个候选、自评、剪枝、回溯。比"多想一会"强在：它把搜索结构显式化、可剪枝可回溯，而单纯让模型内部多想，碰运气成分更大、也无法系统地砍掉烂分支。
  4. Reflexion。两个前提全中：可反复提交（✓ 可重试）、有单元测试（✓ 硬奖励信号）。单纯"失败就重写"不够，因为模型很可能重复同一个错误——它没把"上次为什么挂"显式带进下一次上下文；Reflexion 的自然语言教训正补上这块（HumanEval 91% 的来源）。
  5. 不加。一次 reasoning model 调用即可。难点纯在"想"（读懂 + 压缩表达），没有并行需求、没有可验证搜索、不需要跨尝试学习——命中 §4.3 的"不必费这个劲"。套显式规划器只会和模型内部规划相撞、添乱。
  6. 分段：「读论文、写差异报告」难在想 → 模型内部规划即可；「跑通环境」易反复失败、且有明确"跑没跑通"信号 → 偏向带 reward 的跨尝试重试（Reflexion 式）；「复现多组彼此独立的实验」→ 编排者-工人 / 并行。最危险的相撞点：在"想"占主导的段（读论文）外面硬套一个显式 Plan-and-Execute 图——外层计划和模型自己的拆解打架，正是 §4.3 警告的净负场景。

★ 亲手画图 · 参考答案

轴上从"承诺端"到"适应端"的相对顺序：LLMCompiler（并行 DAG，承诺最重）→ ReWOO（盲蓝图）→ Plan-and-Execute（定计划但可重规划，中段）→ ReAct（每步适应，最右）。轴外三个的"另一根轴"：ToT 在"单计划 ↔ 多计划"那根轴上（它探索多条路径，不是单条计划的承诺早晚问题）；Reflexion 在"单次执行内 ↔ 跨次学习"那根轴上（episode 级，正交于承诺-适应）；reasoning model 不在任何外部轴上——它把规划整个搬进了模型内部，是"要不要外部轴"这个前置问题的另一种答案。
